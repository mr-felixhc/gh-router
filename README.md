# gh-router

**Route the GitHub CLI (`gh`) to the right account automatically, based on the directory you're in.**

No more `gh auth switch`. No more pushing to your work org from a personal repo (or worse, the other way around). Drop into `~/work` and you're your work identity; everywhere else you're you.

```text
~/work/acme-api     $ gh repo view        # → work account (acme org)
~/projects/sideproj $ gh repo view        # → personal account
```

A dependency-free POSIX `sh` wrapper around `gh`. Modular source under `src/`, assembled by `make build` into a single self-contained executable. With no config it is a transparent pass-through.

> 🤖 **Handy for AI coding agents & automation.** Agents like Claude Code or Copilot CLI — and CI scripts — call `gh` non-interactively, across many directories, often in parallel. That's exactly where `gh auth switch`'s global active-account state falls apart. gh-router makes identity a deterministic function of the directory, so automated tooling always acts as the right account with no switching step to get wrong.

---

## Why

`gh` stores credentials per *config directory* (`GH_CONFIG_DIR`) and has `gh auth switch` to flip the **globally active** account. That global switch is the problem:

- It **doesn't persist reliably across non-interactive shells** — scripts, git hooks, CI-on-your-laptop, AI coding agents, editor integrations and parallel terminals each see whatever was last set, which is a recipe for acting as the wrong identity.
- It's **stateful and easy to forget** — the failure mode is silent: a command that resolves the wrong repo, a PR opened from the wrong user, a push rejected with a confusing permissions error.

`gh-router` removes the state. Your *location* determines your *identity*, evaluated fresh on every call — automatically, even inside scripts and hooks.

## How it works

`gh-router` is installed as `gh`, earlier on your `$PATH` than the real binary. On each invocation it:

1. reads `$PWD`,
2. matches it against your rules (the most specific match wins),
3. exports the matching `GH_CONFIG_DIR` (or leaves it unset for your default account),
4. `exec`s the real `gh` with all your arguments untouched.

Because `$PWD` is read at call time, it always follows the directory the command actually runs in. The wrapper runs on *every* `gh` call, so it's deliberately kept to one small, fast script — the modular `src/` files are amalgamated into a single `bin/gh` at build time.

## Install

```sh
git clone https://github.com/mr-felixhc/gh-router
cd gh-router
make install            # builds bin/gh, installs it as ~/.local/bin/gh, scaffolds rules
```

Make sure the install dir is ahead of the real `gh` on `PATH` (the installer warns if not):

```sh
export PATH="$HOME/.local/bin:$PATH"   # in your ~/.bashrc / ~/.zshrc
```

Install elsewhere with `make install PREFIX=/usr/local` (or `BINDIR=...`). Remove with `make uninstall`.

## Configure

Rules live in `~/.config/gh-router/rules`. Format — `<path-glob>` then `<gh-config-dir>`. Among all patterns that match `$PWD`, the **most specific (longest) one wins**, so rule order doesn't matter:

```text
# work account for everything under ~/work, personal everywhere else
$HOME/work        $HOME/.config/gh-work
$HOME/work/*      $HOME/.config/gh-work

# a client inside ~/work on its own account
$HOME/work/acme       $HOME/.config/gh-acme
$HOME/work/acme/*     $HOME/.config/gh-acme

# force the default account for one subtree, overriding a broader rule
$HOME/work/personal-fork    default
```

`~/` and `$HOME` are expanded. Lines starting with `#` are comments. `default` (or `-`) forces `gh`'s normal config.

### Set up an account in a config dir

Each account is just a `gh` config directory you log into once:

```sh
GH_CONFIG_DIR=$HOME/.config/gh-work gh auth login
```

## Verify

```sh
$ cd ~/work/acme-api && GH_ROUTER_DEBUG=1 gh auth status
gh-router: /home/you/work/acme-api -> /home/you/.config/gh-work ($HOME/work/*)
✓ Logged in to github.com account your-work-user

$ cd ~/projects/sideproj && GH_ROUTER_DEBUG=1 gh auth status
gh-router: /home/you/projects/sideproj -> default
✓ Logged in to github.com account your-personal-user
```

`gh --gh-router-version` confirms the wrapper is active.

## Project layout

```
gh-router/
├── src/                 # modular source (edit here)
│   ├── header.sh        #   shebang, doc, defaults
│   ├── resolve.sh       #   locate the real gh
│   ├── rules.sh         #   expand + most-specific match
│   └── main.sh          #   orchestrate + exec
├── tests/routing_test.sh
├── examples/rules.example
├── Makefile             # build / test / lint / install
└── bin/gh               # generated by `make build`
```

## Development

```sh
make build    # assemble src/*.sh -> bin/gh
make test     # build, then run the routing suite (fake gh, asserts routing)
make lint     # shellcheck the assembled script (if installed)
make clean    # remove bin/
```

`bin/gh` is generated — edit the files in `src/` and rebuild.

## Troubleshooting

- **`gh` resolves the wrong repo / "Could not resolve to a Repository":** you're probably in a directory that routes to the wrong account. Run `GH_ROUTER_DEBUG=1 gh ...` to see which config dir was picked — the router keys off `$PWD` at call time.
- **`which gh` doesn't point at the wrapper:** your install dir isn't ahead of the real `gh` on `PATH`. Fix the `PATH` order.
- **Disable temporarily:** call the real binary directly (e.g. `/usr/bin/gh ...`), or `GH_ROUTER_RULES=/dev/null gh ...` for a pass-through.

## Environment

| Variable | Effect |
| --- | --- |
| `GH_ROUTER_RULES` | Override the rules file path |
| `GH_ROUTER_DEBUG` | If non-empty, print the routing decision to stderr |

## Notes & limits

- Path globs and config dirs are whitespace-separated, so paths containing spaces aren't supported (rare for repo roots).
- Routing is by directory only — that's the design. For per-remote routing, `gh`'s native config is the better tool.
